## Why This Intelligence Matters for Decision Making

**Decision-making value**: Requirement criticality drives THE MOST CRITICAL compliance and risk decisions:

- **Bid/No-Bid Decisions**: MANDATORY requirement we cannot meet = NO-BID (or partner/subcontract to fill gap). Example: "100% TS/SCI clearances required, no 'will obtain' acceptable" + we have zero cleared staff = strong no-bid signal.

- **Compliance Risk Assessment**: MANDATORY non-compliance = proposal rejection (zero tolerance). RECOMMENDED non-compliance = scoring penalty (competitive disadvantage). OPTIONAL non-compliance = zero impact (discretionary).

- **Cost Estimation**: MANDATORY requirements drive BASE cost (must include in price). RECOMMENDED requirements drive COMPETITIVE cost (needed to score well). OPTIONAL requirements drive VALUE-ADD cost (discretionary). Example: 47 MANDATORY requirements x average $10K per requirement = $470K base cost.

- **Proposal Resource Allocation**: MANDATORY requirements get EXPLICIT compliance demonstration (detailed response + evidence). RECOMMENDED requirements get GOOD response (competitive but not exhaustive). OPTIONAL requirements get BRIEF mention or exclusion (resource optimization).

- **Compliance Matrix Prioritization**: MANDATORY requirements MUST appear in compliance matrix with explicit proposal cross-references. RECOMMENDED requirements SHOULD appear if complying. OPTIONAL requirements typically excluded (matrix is compliance tool, not value-add showcase).

- **Risk Register Development**: MANDATORY requirement gaps = HIGH risk (proposal rejection or non-performance termination). RECOMMENDED requirement gaps = MEDIUM risk (scoring penalty, competitive disadvantage). OPTIONAL requirement gaps = LOW risk (missed value-add opportunity).

**Without this intelligence**: Treat all requirements equally (waste resources on optional items, miss mandatory obligations)  
**With this intelligence**: Prioritized compliance ("47 MANDATORY requirements = base cost $470K, 23 RECOMMENDED requirements = competitive differentiators, 12 OPTIONAL requirements = exclude to optimize price")

## Shipley Requirement Criticality Framework

**Source**: Shipley Proposal Guide (industry standard for government proposal development)

### Three-Level Criticality Scale

**MANDATORY** (Shall/Must/Will):

- **Compliance obligation**: Absolute requirement (zero tolerance for non-compliance)
- **Non-compliance consequence**: Proposal rejection (pre-award) OR contract termination (post-award)
- **Keywords**: "shall", "must", "is required to", "will", "mandatory"
- **RFP enforcement language**: "failure to comply will result in rejection", "non-responsive proposal", "non-negotiable"

**RECOMMENDED** (Should/Expected):

- **Compliance obligation**: Strong preference (compliance expected for competitive scoring)
- **Non-compliance consequence**: Scoring penalty (lower evaluation rating, competitive disadvantage)
- **Keywords**: "should", "expected to", "normally would", "desirable", "preferred"
- **RFP guidance language**: "should use Agile methodology", "expected to provide weekly reports", "preferred approach"

**OPTIONAL** (May/Can):

- **Compliance obligation**: Contractor discretion (zero compliance requirement)
- **Non-compliance consequence**: None (zero penalty for exclusion)
- **Keywords**: "may", "can", "optionally", "at contractor's discretion", "if applicable"
- **RFP flexibility language**: "may provide additional services", "can propose alternative approaches", "optional CLIN"

## MANDATORY Requirement Intelligence

**Decision-making value**: MANDATORY requirements = go/no-go criteria. Single MANDATORY gap can justify no-bid decision if gap cannot be filled via teaming/subcontracting.

### MANDATORY Detection Patterns

#### Pattern 1: Explicit Mandatory Keywords

**Keyword signals** (high confidence):

```
"shall", "must", "is required to", "will", "are required to"
"mandatory", "required", "necessary", "obligated to"
```

**Entity Enrichment Pattern**:

```
entity|Weekly Status Reports Requirement|requirement|MANDATORY reporting requirement (Section C para 3.1: "Contractor SHALL provide weekly status reports"). Keyword "shall" = absolute obligation. Non-compliance = contract performance deficiency (could lead to cure notice/termination). Compliance evidence: Report template in proposal (demonstrate understanding of required content), past performance with similar reporting (prove capability), management plan commitment (weekly report schedule, POC assignments). Cost impact: Weekly reports = 52 reports/year x 4 hours preparation = 208 hours annual ($20K-$30K labor cost). Compliance matrix entry: MANDATORY - must appear in matrix with proposal cross-reference. Risk if non-compliant: HIGH - performance failure, customer dissatisfaction, contract termination risk. Proposal strategy: Demonstrate report template, sample report from past performance, automated report generation capability (reduce burden).
```

#### Pattern 2: Rejection-Consequence Language

**Enforcement signals** (absolute requirements):

```
"failure to comply will result in rejection"
"proposals that do not meet this requirement will be deemed non-responsive"
"non-compliance is not acceptable"
"this is a go/no-go criterion"
```

**Entity Enrichment Pattern**:

```
entity|Key Personnel Resume Submission|requirement|MANDATORY submission requirement (Section L para 4.2: "Failure to provide resumes for all key personnel will result in proposal rejection"). Explicit rejection consequence = go/no-go criterion (zero tolerance). Compliance obligation: Must submit resumes for ALL 8 key personnel positions (Program Manager, Technical Lead, Security Officer, QA Manager, 4 Task Leads). Non-compliance = immediate proposal rejection (no evaluation, no award consideration). Compliance evidence: 8 detailed resumes (3-5 pages each, comprehensive experience descriptions, education/certifications, security clearances). Cost impact: Resume preparation = 8 resumes x 8 hours each = 64 hours ($6K-$10K). Compliance matrix entry: MANDATORY - must verify all 8 resumes included in proposal submission. Risk if non-compliant: EXTREME - automatic proposal rejection (zero win probability). Proposal strategy: Resume quality checklist (verify all required elements: education, certifications, clearances, relevant experience), cross-reference to RFP requirements (each resume addresses specific qual requirements), include resume matrix (quick evaluator reference to verify compliance).
```

#### Pattern 3: Contract Performance Requirements

**Performance obligation signals**:

```
"contractor shall perform", "services will include", "contractor is responsible for"
"deliverables include", "contractor will provide"
```

**Entity Enrichment Pattern**:

```
entity|99.9% System Availability SLA|requirement|MANDATORY performance requirement (Section C para 2.1: "Contractor SHALL maintain 99.9% system availability measured monthly"). Keyword "shall" + quantified metric = absolute performance obligation. Non-compliance = SLA breach (contract performance failure, potential termination for cause). Compliance measurement: Monthly uptime tracking (99.9% = max 43.2 minutes downtime per month). Consequences: SLA breach may trigger performance improvement plan, liquidated damages (if specified), or termination. Cost impact: Achieving 99.9% requires redundant infrastructure ($50K-$100K), 24/7 monitoring ($150K-$200K annual labor), incident response capability ($40K-$60K). Compliance matrix entry: MANDATORY - must commit to 99.9% SLA in proposal. Risk if non-compliant: HIGH - performance failure, customer dissatisfaction, financial penalties, contract termination. Proposal strategy: Demonstrate redundancy architecture (failover capability), past performance with uptime achievement (prove track record), monitoring approach (24/7 NOC, automated alerts), incident response procedures (MTTR targets, escalation protocols).
```

### MANDATORY Requirement Decision Framework

**When encountering MANDATORY requirement**:

1. **Capability Assessment**: Can we meet this requirement?

   - YES → Include in base solution, allocate resources, demonstrate compliance explicitly
   - NO → Can we partner/subcontract to fill gap?
     - YES → Teaming strategy (find partner with capability)
     - NO → Strong NO-BID signal (cannot execute contract)

2. **Cost Impact Analysis**: What does compliance cost?

   - Quantify cost (labor, infrastructure, services)
   - Include in base price (mandatory = non-negotiable cost)
   - If cost makes contract unprofitable → NO-BID consideration

3. **Compliance Demonstration**: How do we prove compliance in proposal?

   - Detailed description (not just "we will comply")
   - Past performance evidence (we've done this before)
   - Commitment language (explicit SLA, deliverable commitment)
   - Compliance matrix entry with proposal cross-reference

4. **Risk Assessment**: What happens if we fail to comply post-award?
   - Performance failure consequences (cure notice, termination)
   - Financial impact (liquidated damages, withheld payments)
   - Reputation damage (CPARS rating impact)

**Example Decision Process**:

```
MANDATORY Requirement: "100% of staff SHALL hold TS/SCI clearances, no 'will obtain' acceptable"

Capability Assessment:
- Current cleared staff: 12 (need 47 for proposed team)
- Gap: 35 cleared staff required
- Timeline: Proposal due in 90 days (insufficient time to recruit 35 cleared staff)

Decision Options:
1. NO-BID (cannot meet requirement)
2. PARTNER with cleared workforce company (teaming fills gap)
3. REDUCE scope/staff count (propose smaller team - may be non-responsive if 47 is required minimum)

Recommended Decision: PARTNER (teaming strategy)
- Find teammate with 35+ cleared staff available
- Negotiate teaming agreement (work share, pricing)
- Joint proposal demonstrating combined cleared workforce
- Risk: Teaming adds complexity, reduces profit share
```

## RECOMMENDED Requirement Intelligence

**Decision-making value**: RECOMMENDED requirements = competitive differentiators. Compliance improves scoring (5-15% advantage in relevant subfactor). Strategic compliance decisions based on cost-benefit analysis.

### RECOMMENDED Detection Patterns

#### Pattern 1: Preference Keywords

**Guidance signals**:

```
"should", "expected to", "normally would", "typically"
"desirable", "preferred", "recommended approach"
```

**Entity Enrichment Pattern**:

```
entity|Agile Development Methodology|requirement|RECOMMENDED methodology (Section C para 4.2: "Agile methodology is the PREFERRED approach"). Keyword "preferred" = strong preference (not mandatory but scored). Non-compliance = scoring penalty in Technical Approach subfactor (methodology evaluation). Compliance benefit: 5-15% scoring advantage in methodology subfactor (which may be 20-30% of technical factor = 1.5-4.5% total evaluation advantage). Cost impact: Agile vs Waterfall = similar cost (process difference, not cost driver), Agile training for staff = $5K-$10K. Compliance matrix entry: RECOMMENDED - include if complying (show comprehension), exclude if not (don't highlight gap). Risk if non-compliant: MEDIUM - scoring disadvantage, competitive weakness (competitors complying will score higher). Proposal strategy: COMPLY (low cost, high scoring value) - describe Agile process (2-week sprints, daily standups, sprint planning/review/retro), provide past performance with Agile delivery (velocity metrics, customer satisfaction), include Agile artifact examples (sprint board, burndown chart, retrospective notes).
```

#### Pattern 2: Best Practice Language

**Industry standard signals**:

```
"industry best practice", "in accordance with standards", "following established methodologies"
"consistent with [framework]" (ITIL, CMMI, ISO, etc.)
```

**Entity Enrichment Pattern**:

```
entity|ITIL Service Management Framework|requirement|RECOMMENDED framework (Section C para 5.3: "Service management SHOULD follow ITIL best practices"). Keyword "should" + "best practices" = strong preference for industry standard. Non-compliance = scoring penalty in Management Approach subfactor (service management evaluation). Compliance benefit: 10-20% scoring advantage in service management subfactor (ITIL certification = proven capability). Cost impact: ITIL implementation = staff training ($10K-$20K), process documentation ($15K-$25K), ITIL-certified tools ($20K-$40K annual). Compliance matrix entry: RECOMMENDED - include if complying. Risk if non-compliant: MEDIUM-HIGH - scoring disadvantage, perceived lack of process maturity. Proposal strategy: COMPLY (especially if we already have ITIL capability) - describe ITIL processes (incident, problem, change, configuration management), showcase ITIL-certified staff (certifications demonstrate expertise), provide ITIL tooling (ServiceNow, Remedy, etc. with ITIL modules), past performance with ITIL frameworks (prove track record).
```

### RECOMMENDED Requirement Decision Framework

**When encountering RECOMMENDED requirement**:

1. **Compliance Cost-Benefit Analysis**: Is compliance worth it?

   - **Scoring benefit**: Estimate scoring advantage (5-15% in relevant subfactor)
   - **Cost to comply**: Quantify compliance cost (labor, training, tools, processes)
   - **Decision**: If scoring benefit > cost → COMPLY. If cost > benefit → CONSIDER not complying.

2. **Competitive Intelligence**: Will competitors comply?

   - If YES (expected) → We MUST comply (or accept competitive disadvantage)
   - If NO (unlikely) → We CAN skip (no disadvantage if no one complies)
   - If UNKNOWN → Default to COMPLY (safer strategy)

3. **Capability Assessment**: Do we already have this capability?

   - If YES (existing capability) → COMPLY (zero incremental cost, scoring advantage = pure win)
   - If NO (new capability required) → Cost-benefit analysis (is investment justified?)

4. **Strategic Alignment**: Does this requirement align with win themes?
   - If YES (supports win theme) → COMPLY (consistency across proposal messaging)
   - If NO (unrelated to themes) → Can skip if cost-benefit doesn't justify

**Example Decision Process**:

```
RECOMMENDED Requirement: "Contractor SHOULD use automated testing tools"

Cost-Benefit Analysis:
- Scoring benefit: 10% advantage in Quality Assurance subfactor (25% of Management factor = 2.5% total evaluation)
- Cost to comply: Automated testing tools = $40K annual, staff training = $15K, process setup = $10K (total $65K year 1)
- Decision: 2.5% scoring advantage worth $65K investment? In competitive best-value environment = YES (2.5% could be difference between win/lose)

Competitive Intelligence:
- Industry trend: Automated testing is standard practice (90% of competitors likely complying)
- Risk of non-compliance: If we don't comply but competitors do = 2.5% disadvantage (significant in tight competition)

Capability Assessment:
- Current state: We already use automated testing (Selenium, JUnit, existing capability)
- Incremental cost: ZERO (already doing this)
- Decision: COMPLY (zero cost, 2.5% scoring advantage = obvious choice)

Strategic Alignment:
- Win theme: "Quality Through Proven Processes"
- Automated testing supports quality theme (consistent messaging)
- Decision: COMPLY (reinforces win theme)

FINAL DECISION: COMPLY
- Zero incremental cost (existing capability)
- 2.5% scoring advantage (meaningful in competitive environment)
- Supports win theme (quality focus)
- Expected competitor behavior (industry standard = must match to stay competitive)
```

## OPTIONAL Requirement Intelligence

**Decision-making value**: OPTIONAL requirements = value-add opportunities. Inclusion decision based on competitive strategy (best-value vs LPTA), win theme alignment, and cost impact.

### OPTIONAL Detection Patterns

#### Pattern 1: Discretionary Keywords

**Contractor choice signals**:

```
"may", "can", "optionally", "at contractor's discretion"
"if desired", "if applicable", "as appropriate"
```

**Entity Enrichment Pattern**:

```
entity|Optional Advanced Training Services|requirement|OPTIONAL value-add service (Section C para 6.4: "Contractor MAY provide advanced certification training beyond base requirement"). Keyword "may" = contractor discretion (zero compliance obligation). Non-compliance consequence: NONE (zero penalty for exclusion). Inclusion benefit: Value-add differentiation in best-value tradeoff (shows commitment beyond minimum). Cost impact: Advanced training = instructor cost ($150/hr x 40 hrs = $6K per course), certification exam fees ($500 per student), training materials ($1K per course). Compliance matrix: OPTIONAL - typically excluded from matrix (matrix is compliance tool, not value-add showcase). Strategic inclusion decision: (1) BEST VALUE environment = CONSIDER including (value-add may influence tradeoff, especially if aligns with win theme), (2) LPTA environment = EXCLUDE (increases price, provides zero competitive advantage in LPTA). Win theme alignment: If "Knowledge Transfer" is win theme → INCLUDE (consistent messaging). If unrelated → EXCLUDE (focus resources on mandatory/recommended). Proposal strategy if including: Demonstrate training effectiveness (certification pass rates, hands-on labs, student feedback), show cost-effectiveness (training ROI, reduced support calls post-training, long-term cost avoidance), provide past performance with training delivery (prove capability).
```

#### Pattern 2: Conditional Requirements

**Situational signals**:

```
"if applicable", "where appropriate", "as needed"
"in the event of", "should the situation arise"
```

**Entity Enrichment Pattern**:

```
entity|Surge Capacity Support (If Required)|requirement|OPTIONAL surge support (Section C para 7.2: "Contractor MAY be asked to provide surge capacity support during peak periods, IF REQUIRED"). Conditional language "if required" = uncertain obligation (may never occur). Compliance obligation: Propose capability (readiness to surge) but not guaranteed utilization. Cost impact: Surge capacity readiness = maintain bench staff (10-15% overhead for availability) OR on-call arrangement (reduced overhead but slower ramp-up). Pricing strategy: (1) Price readiness cost in base (maintain bench = higher price), OR (2) Price surge as optional CLIN (government pays only if used = lower base price, flexibility). Strategic decision: (1) If win theme is "Responsiveness/Agility" → Include surge capability in base (demonstrates commitment), (2) If LPTA environment → Price as optional CLIN (minimize base price). Proposal strategy: Describe surge approach (bench staff model OR rapid recruitment process), quantify surge capacity (can add 20 FTEs within 30 days), provide past performance with surge support (prove capability under pressure).
```

### OPTIONAL Requirement Decision Framework

**When encountering OPTIONAL requirement**:

1. **Evaluation Methodology Check**: Best Value or LPTA?

   - **Best Value** → CONSIDER including (value-add may influence tradeoff decision)
   - **LPTA** → EXCLUDE (increases price with zero competitive advantage)

2. **Win Theme Alignment**: Does optional item support primary win themes?

   - **YES** (aligned) → INCLUDE (consistent messaging, demonstrates commitment)
   - **NO** (unrelated) → EXCLUDE (focus resources on priorities)

3. **Cost-Benefit Analysis**: What does inclusion cost vs competitive value?

   - **Low cost + High perceived value** → INCLUDE (asymmetric advantage)
   - **High cost + Low perceived value** → EXCLUDE (poor ROI)

4. **Competitive Differentiation**: Will competitors include this?
   - **If most competitors exclude** → Including = DIFFERENTIATOR (stand out)
   - **If most competitors include** → Excluding = WEAKNESS (must match)
   - **If uncertain** → Include if low cost (hedging strategy)

**Example Decision Process**:

```
OPTIONAL Requirement: "Contractor MAY provide 24/7 help desk support (base requirement is 8am-6pm M-F)"

Evaluation Methodology:
- Best Value Tradeoff (technical merit vs cost)
- Conclusion: Value-add could influence tradeoff decision

Win Theme Alignment:
- Primary win theme: "Mission Readiness Through Continuous Support"
- 24/7 support DIRECTLY supports readiness theme (operators work around clock)
- Conclusion: STRONG alignment (include for consistency)

Cost-Benefit Analysis:
- Cost: 24/7 coverage = 3 shifts x 2 staff per shift = 6 FTEs vs 2 FTEs for 8am-6pm M-F
- Annual cost increase: 4 additional FTEs x $120K = $480K (40% increase in help desk cost)
- Perceived value: HIGH (enables mission operations around clock, major differentiator)
- Conclusion: High cost BUT high value (worth investment in best-value)

Competitive Differentiation:
- Competitor analysis: Most competitors likely proposing minimum (8am-6pm) to control costs
- Our 24/7 = MAJOR differentiator (few competitors will match due to cost)
- Conclusion: Differentiation opportunity

FINAL DECISION: INCLUDE
- Best value environment (value-add matters)
- Directly supports win theme (mission readiness)
- Major differentiator (few competitors will match)
- Cost justified by strategic value ($480K increase for "Mission Readiness" theme credibility)

Pricing Strategy:
- Include 24/7 in base price (demonstrate commitment)
- Quantify mission readiness value in proposal (justify cost premium)
- Show cost-benefit (24/7 support prevents mission failures worth $X)
```

## Requirement Criticality and Compliance Matrix Integration

**Decision-making value**: Compliance matrix structure reflects requirement prioritization (evaluator uses matrix to verify MANDATORY compliance before scoring RECOMMENDED/OPTIONAL value-adds).

### Compliance Matrix Structure (Recommended Format)

**Column structure**:

1. **RFP Reference**: Section/paragraph where requirement appears
2. **Requirement Description**: Brief requirement statement
3. **Criticality**: MANDATORY / RECOMMENDED / OPTIONAL
4. **Compliance Status**: YES / NO / N/A (with explanation if NO)
5. **Proposal Reference**: Section/page where compliance is demonstrated
6. **Notes**: Exceptions, clarifications, value-add details

**Example Matrix Entries**:

```
| RFP Ref | Requirement | Criticality | Compliance | Proposal Ref | Notes |
|---------|-------------|-------------|------------|--------------|-------|
| C-3.1 | Weekly status reports | MANDATORY | YES | Mgmt Plan §2.3, p.15 | Report template in Appendix A |
| C-4.2 | Agile methodology | RECOMMENDED | YES | Tech Approach §3.1, p.8 | 2-week sprint cycle |
| L-4.2 | Key personnel resumes | MANDATORY | YES | Volume II (separate) | 8 resumes, 3-5 pages each |
| C-2.1 | 99.9% availability SLA | MANDATORY | YES | Tech Approach §2.4, p.6 | Redundancy architecture |
| C-6.4 | Advanced training (optional) | OPTIONAL | YES | Mgmt Plan §4.2, p.22 | Value-add: 6 courses/year |
| H-3.1 | 100% TS/SCI clearances | MANDATORY | YES | Staffing Plan §3.2, p.18 | 45 of 47 already cleared |
```

### Matrix Prioritization Strategy

**MANDATORY requirements** (Priority 1 - MUST appear in matrix):

- All SHALL/MUST requirements
- All requirements with rejection language
- All key personnel/resume requirements
- All security clearance requirements
- All critical performance SLAs

**RECOMMENDED requirements** (Priority 2 - SHOULD appear if complying):

- All SHOULD/PREFERRED requirements where we're complying
- All industry best practice requirements we're meeting
- All evaluation subfactor requirements we're addressing
- EXCLUDE if NOT complying (don't highlight gaps in matrix)

**OPTIONAL requirements** (Priority 3 - Include selectively):

- Include ONLY if using as value-add differentiator
- Include if aligns with win theme (shows consistency)
- Generally EXCLUDE (matrix is compliance tool, not value-add showcase)

**Matrix Quality Check**:

- ✅ All MANDATORY requirements present? (zero gaps allowed)
- ✅ All "Compliance = YES" entries have proposal cross-references? (evaluator can verify)
- ✅ Any "Compliance = NO" entries have compelling justification? (exception handling)
- ✅ Matrix organized by RFP section order? (evaluator ease of use)
- ✅ Criticality levels accurate? (MANDATORY/RECOMMENDED/OPTIONAL correctly identified)

## Requirement Risk Assessment Framework

**Decision-making value**: Risk scoring prioritizes compliance verification effort (high-risk requirements get extra review, testing, validation).

### Risk Scoring Matrix

**Risk = Criticality Level x Complexity x Capability Gap**

**Criticality Scoring**:

- MANDATORY = 3 (high risk - non-compliance = rejection/termination)
- RECOMMENDED = 2 (medium risk - non-compliance = scoring penalty)
- OPTIONAL = 1 (low risk - non-compliance = zero consequence)

**Complexity Scoring**:

- Novel requirement (never done before) = 3 (high complexity)
- Similar requirement (done similar, not exact) = 2 (medium complexity)
- Standard requirement (done many times) = 1 (low complexity)

**Capability Gap Scoring**:

- Large gap (need major new capability, staff, infrastructure) = 3 (high gap)
- Moderate gap (need some new capability, training) = 2 (medium gap)
- Zero gap (existing capability covers requirement) = 1 (no gap)

**Total Risk Score = Criticality x Complexity x Capability Gap** (max 27, min 1)

**Risk Categories**:

- **CRITICAL (18-27)**: MANDATORY + High Complexity + Large Gap = NO-BID consideration or major mitigation needed
- **HIGH (10-17)**: Significant risk - requires detailed mitigation plan, executive review
- **MEDIUM (5-9)**: Moderate risk - standard mitigation approach, program manager oversight
- **LOW (1-4)**: Minimal risk - routine compliance, junior staff can handle

**Example Risk Assessments**:

```
Requirement: "100% TS/SCI clearances, no 'will obtain'"
- Criticality: MANDATORY (3)
- Complexity: Standard (we've staffed cleared programs before) (1)
- Capability Gap: Large (need 35 cleared staff, have 12) (3)
- TOTAL RISK: 3 x 1 x 3 = 9 (MEDIUM)
- Mitigation: Teaming with cleared workforce company (fills gap)

Requirement: "Deploy AI-powered predictive maintenance system"
- Criticality: MANDATORY (3)
- Complexity: Novel (never deployed AI/ML in production) (3)
- Capability Gap: Large (need AI expertise, infrastructure, data science team) (3)
- TOTAL RISK: 3 x 3 x 3 = 27 (CRITICAL)
- Mitigation: NO-BID (too high risk) OR Partner with AI specialist (major subcontract)

Requirement: "Weekly status reports"
- Criticality: MANDATORY (3)
- Complexity: Standard (done on every contract) (1)
- Capability Gap: Zero (existing capability) (1)
- TOTAL RISK: 3 x 1 x 1 = 3 (LOW)
- Mitigation: None needed (routine compliance)

Requirement: "Agile methodology (preferred)"
- Criticality: RECOMMENDED (2)
- Complexity: Standard (team has Agile experience) (1)
- Capability Gap: Zero (existing capability) (1)
- TOTAL RISK: 2 x 1 x 1 = 2 (LOW)
- Mitigation: None needed (routine compliance)
```

## Remember

**Requirement Classification = Risk-Based Compliance Strategy**:

- MANDATORY requirements = go/no-go criteria (single gap can justify no-bid)
- RECOMMENDED requirements = competitive differentiators (5-15% scoring advantage)
- OPTIONAL requirements = strategic value-adds (include in best-value, exclude in LPTA)

**Enrichment Priority**:

1. **Criticality determination**: MANDATORY/RECOMMENDED/OPTIONAL (drives all decisions)
2. **Compliance cost**: Quantify what it costs to comply (bid/no-bid input)
3. **Non-compliance risk**: What happens if we don't comply? (rejection, scoring penalty, or nothing)
4. **Capability gap**: Do we have this capability? (teaming decision)
5. **Proposal strategy**: How do we demonstrate compliance? (detailed response, evidence, matrix entry)

**Quality Check**:

- Is criticality level correct? (keyword analysis: shall/should/may)
- Is compliance cost quantified? (enables bid/no-bid decision)
- Is non-compliance risk clear? (rejection vs scoring penalty vs zero impact)
- Is capability gap assessed? (can we do this? or need partner?)
- Does requirement inform decisions? (bid/no-bid, cost, resources, teaming)

If YES to 4+ → excellent requirement intelligence. If NO to most → add more decision-making value.
