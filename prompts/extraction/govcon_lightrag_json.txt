PART A.2: NAMING & CONSISTENCY RULES (LightRAG Entity Merging Optimization)
================================================================================
These rules ensure entities extracted from different chunks merge correctly and 
relationships map accurately across the knowledge graph.

A.2.1 FULL CONTEXTUAL ENTITY NAMING
-----------------------------------
Entity names MUST include both identifier AND descriptive title for uniqueness 
and searchability. Generic identifiers alone are insufficient.

SECTIONS & APPENDICES:
✅ GOOD: "Appendix H Al Dhafra Air Base (ADAB), UAE"
✅ GOOD: "Section L Proposal Preparation Instructions"
✅ GOOD: "Section M Evaluation Factors for Award"
❌ BAD:  "Appendix H" (ambiguous - could be any document's Appendix H)
❌ BAD:  "Section L" (too generic)

CLAUSES:
✅ GOOD: "FAR 52.212-4 Contract Terms Commercial Items"
✅ GOOD: "DFARS 252.204-7012 Safeguarding Covered Defense Information"
❌ BAD:  "FAR 52.212-4" (missing title context)
❌ BAD:  "Cybersecurity Clause" (missing identifier)

CDRLs & DELIVERABLES:
✅ GOOD: "CDRL A002 Task Order Management Plan"
✅ GOOD: "CDRL A014 Quality Control Plan"
❌ BAD:  "CDRL A002" (missing deliverable name)
❌ BAD:  "Monthly Status Report" (missing CDRL identifier)

REQUIREMENTS:
✅ GOOD: "F.2.1 CRP Customer Service Coverage Requirement"
✅ GOOD: "C.3.2 Network Operations Support Requirement"
❌ BAD:  "Customer Service Requirement" (missing section reference)

A.2.2 NAMING CONSISTENCY ACROSS CHUNKS
--------------------------------------
The SAME entity must have the SAME name every time it appears, regardless of 
which chunk it's extracted from. This enables correct entity merging.

RULES:
- Use Title Case consistently for all entity names
- Preserve exact identifiers (section numbers, clause numbers, CDRL IDs)
- If an entity was named "Appendix H Al Dhafra Air Base (ADAB), UAE" in one 
  chunk, use that EXACT name in all subsequent references
- Do NOT create variations like "APPENDIX H", "Appendix H - ADAB", "App H"

A.2.3 THIRD-PERSON OBJECTIVE DESCRIPTIONS
-----------------------------------------
All entity descriptions must be written in third person. Avoid pronouns that 
create ambiguity or lose context when the description is read standalone.

✅ GOOD: "The Contractor shall provide 24/7 support for CRP operations."
✅ GOOD: "Appendix H defines site-specific requirements for Al Dhafra Air Base."
❌ BAD:  "This section defines the requirements." (what section?)
❌ BAD:  "We shall provide support." (who is we?)
❌ BAD:  "As stated above, the contractor..." (above where?)

AVOID THESE PRONOUNS: this, that, these, those, it, our, we, I, you, he, she, 
they, above, below, the following, as mentioned

A.2.4 RELATIONSHIP HYGIENE
--------------------------
- Avoid duplicate relationships (A→B and B→A for undirected relationships)
- Use consistent entity names in source_entity and target_entity fields
- Entity names in relationships MUST exactly match extracted entity names

A.2.5 CANONICAL ENTITY FORM RULE
--------------------------------
When the same logical entity appears with different name variations across 
chunks, you MUST normalize to a single canonical form. This prevents duplicate 
nodes in the knowledge graph.

CANONICAL FORM PATTERN: "(Identifier) (Full-Descriptive-Title)"

RECOGNITION RULES:
- If two names share the same identifier (e.g., "Appendix G"), they are the 
  SAME entity regardless of how the descriptive portion varies
- If two names share the same acronym in parentheses, they are the SAME entity
- If one name is a substring/abbreviation of another, use the LONGER form

EXAMPLES OF MERGING TO CANONICAL FORM:
  "Appendix G" + "Appendix G AUAB" + "Appendix G Al Udeid Air Base" 
  → Canonical: "Appendix G Al Udeid Air Base (AUAB)"
  
  "CDRL A002" + "CDRL A002 Task Order Management Plan"
  → Canonical: "CDRL A002 Task Order Management Plan"
  
  "Section L" + "Section L Proposal Preparation Instructions"
  → Canonical: "Section L Proposal Preparation Instructions"

RULE: When you encounter a short-form reference (e.g., "Appendix G"), check if 
you have already extracted a longer canonical form. If yes, use the canonical 
form. If this is the first occurrence, extract with maximum available context.

A.2.6 HIERARCHICAL SIBLING RELATIONSHIPS
----------------------------------------
Document sections that share the same structural level are SIBLINGS, not a 
parent-child chain. This rule prevents incorrect CHILD_OF relationship chains.

IMPORTANT: Siblings being structurally independent does NOT mean they are 
unrelated. Sibling sections often have rich cross-references and content 
connections that SHOULD be extracted as relationships.

SIBLING PATTERN RECOGNITION:
- Lettered sequences at same level: A, B, C, D... are siblings
- Numbered sequences at same level: 1, 2, 3, 4... are siblings
- Roman numeral sequences: I, II, III, IV... are siblings
- Any series of top-level document divisions are siblings to each other
- Attachments, exhibits, appendices, annexes at same level are siblings

CORRECT HIERARCHY PATTERN:
  Parent Document
  ├── Division X (sibling to Y and Z)
  ├── Division Y (sibling to X and Z)
  │   ├── Y.1 Sub-section (child of Division Y)
  │   └── Y.2 Sub-section (child of Division Y)
  └── Division Z (sibling to X and Y)

INCORRECT PATTERN (DO NOT DO):
  Division X → CHILD_OF → Division Y → CHILD_OF → Division Z  ❌

WHEN TO USE CHILD_OF:
- Sub-sections within a parent section (3.1 is CHILD_OF Section 3)
- Exhibits/tables within a parent attachment
- Numbered paragraphs within a clause or article
- Any content explicitly nested under a parent heading

CROSS-REFERENCES BETWEEN SIBLINGS (ENCOURAGED):
Sibling sections often reference each other's content. Extract these as 
relationships using appropriate types like REFERENCES, MAPS_TO, EVALUATED_BY, 
SUPPORTS, INFORMS, SPECIFIES, CONSTRAINS, etc.

PATTERN-BASED CROSS-REFERENCE EXAMPLES:
- Submission instructions → MAPS_TO → Evaluation criteria
- Workload/volume data → INFORMS → Performance requirements
- Deliverable specifications → REFERENCES → Data format definitions
- Evaluation factors → EVALUATED_BY → Instructions in another section
- Technical requirements → CONSTRAINED_BY → Terms and conditions

These cross-section connections are CRITICAL for traceability and compliance 
mapping. Extract them whenever the text indicates a relationship between 
content in structurally separate sections, regardless of document format.

PART B: QUANTITATIVE DETAIL PRESERVATION (Critical for BOE Development)
================================================================================
Quantitative details serve multiple purposes across the capture team:
- Cost Estimators: labor calculations, BOE development, pricing models
- Proposal Writers: specific compliance language, technical accuracy
- Capture Managers: scope understanding, risk assessment
- Contracts Managers: service levels for negotiation, performance metrics

ALWAYS extract these VERBATIM - never summarize, generalize, or omit numbers!

SERVICE RATES - Extract exact rates with context:
- "X customer(s) per minute/hour" → preserve exactly
- "X transaction(s) per shift" → preserve exactly
- Distinguish: normal operations vs peak times vs events
- Include time windows: (0500-0700, 1100-1300, etc.)
Example: "one (1) customer per minute during normal operations and three (3) customers per minute during peak times (0500-0700, 1100-1300, 1900-2100, 2300-0100)"

FREQUENCIES - Extract with multipliers:
- "X times per year/month/week/day"
- "estimated X occurrences annually"
- "average X-Y [deliveries/visits/events]"
- NEVER generalize "100 times per year" to "frequent" or "regular"
Example: "estimated to occur 100 times per year" → preserve "100 times per year"

DOLLAR VOLUMES - Preserve ranges:
- "$X-Y per night/month/transaction"
- "up to $X during [condition]"
- "between $X and $Y"
Example: "sells between $2,000 and $5,000 per night, can exceed $10,000 during events"

QUANTITIES & COUNTS:
- Equipment: "X units", "Y registers", "Z refrigeration units"
- Personnel: "X staff", "Y FTEs required", "Z supervisors"
- Inventory: "X items", "Y tubs annually", "Z cases per week"
Example: "750 tubs of 700-count wipes annually" → preserve BOTH numbers

TIME RANGES - Never drop time specifications:
- Operating hours: "1830-0200", "0700-1900"
- Peak periods: "0500-0700, 1100-1300, 1900-2100, 2300-0100"
- Response times: "within X minutes/hours", "NLT X days"
- Duration: "2-3 calendar days", "7 days advance notice"

COVERAGE SPECIFICATIONS:
- "24/7", "24 hours/day, 7 days/week"
- Population served: "X daily customers, up to Y during surges"
Example: "1,600 daily customers (up to 4,000 during rotations)"

EXTRACTION ANTI-PATTERNS (DO NOT DO):
❌ "Contractor shall provide customer service" (loses service rate)
❌ "Regular events" (loses "100 times per year")
❌ "Sales operations" (loses "$2,000-5,000 per night")
❌ "Equipment available" (loses "8 refrigeration units")

PART B.2: CRITICAL TABLE EXTRACTION RULES (Equipment & Maintenance Data)
================================================================================

**CRITICAL: Tables contain high-value workload data that MUST be extracted as structured entities.**

Tables are NOT just formatting - they contain critical workload drivers for BOE development.
The LLM must recognize table structures and extract granular data, not summaries.

B.2.1 EQUIPMENT INVENTORY TABLES (e.g., "Table H.2 Appliance Listing")
----------------------------------------------------------------------
When processing equipment inventory tables:

1. **Extract EACH building as a LOCATION entity** with equipment counts
   - Entity: "Building 2959 Laundry Facility" (location)
   - Include counts in description: "69 washers, 69 dryers (138 total units)"

2. **Extract equipment aggregates as EQUIPMENT entities** by maintenance schedule
   - Entity: "Weekly Maintenance Equipment Inventory" (equipment)
   - Include totals: "370 washers, 399 dryers (769 total units)"

3. **Create LOCATION→EQUIPMENT relationships** using HAS_EQUIPMENT
   - Building 2959 --HAS_EQUIPMENT--> Weekly Maintenance Equipment Inventory

4. **Preserve maintenance schedule context** (Weekly, Monthly, On-Call)
   - Include schedule in entity description for FTE calculation

5. **Extract ALL rows, not just summaries** - building-level data is critical for workload estimation
   - Each building = separate LOCATION entity
   - Summary totals = EQUIPMENT entity aggregates

B.2.2 MAINTENANCE SCHEDULE TABLES (e.g., "Table H.1 Preventive Maintenance")
----------------------------------------------------------------------------
When processing maintenance schedule tables:

1. **Extract maintenance tasks as REQUIREMENT entities**
   - Include frequency in labor_drivers: "weekly visual inspection", "monthly lubrication"

2. **Capture frequencies explicitly**:
   - Weekly tasks → labor_drivers: ["weekly"]
   - Monthly tasks → labor_drivers: ["monthly"]
   - Quarterly tasks → labor_drivers: ["quarterly"]
   - Annual/Semiannual → labor_drivers: ["2x per year"] or ["annually"]

3. **Extract equipment types as EQUIPMENT entities**
   - "Treadmills" (equipment)
   - "Stationary Cycles" (equipment)

4. **Create APPLIES_TO relationships between requirements and equipment**
   - Treadmill Preventive Maintenance --APPLIES_TO--> Treadmills

B.2.3 TABLE EXTRACTION PRIORITIES
---------------------------------
✅ MUST DO:
- Extract ALL rows with quantities (not just totals/summaries)
- Preserve building numbers, equipment counts, and schedules
- Link tables to their parent appendix/section using CHILD_OF
- Create relationships between locations, equipment, and requirements

❌ MUST NOT DO:
- Do NOT extract tables as generic "concept" entities
- Do NOT summarize table data - extract each location/equipment row
- Do NOT lose building-level granularity
- Do NOT merge disparate maintenance schedules into single entities

ANTI-PATTERN (WRONG):
{{"name": "Laundry Equipment", "type": "concept", "description": "Various laundry equipment across multiple buildings."}}

CORRECT PATTERN:
{{"name": "Building 2959 Laundry Facility", "type": "location", "description": "Building 2959. Equipment counts: 69 washers, 69 dryers (138 total units). Maintenance schedule: Weekly."}}
{{"name": "Building 3880 Laundry Facility", "type": "location", "description": "Building 3880. Equipment counts: 44 washers, 50 dryers (94 total units). Maintenance schedule: Weekly."}}
{{"name": "Weekly Maintenance Equipment Inventory", "type": "equipment", "description": "Weekly maintenance schedule equipment. Total: 370 washers, 399 dryers (769 total units)."}}

B.2.4 TABLE TYPE-SPECIFIC GUIDANCE (MinerU Table Processing)
------------------------------------------------------------
MinerU converts tables to structured text. Use these guidelines for different table types:

**Evaluation Matrices (Section M):**
- Extract relationships between factors and their evaluation criteria
- Look for weight/point distributions across subfactors
- Create CHILD_OF relationships for subfactor hierarchy

**CDRL Schedules (Section J):**
- Extract relationships between deliverables and their requirements
- Capture due dates, frequencies, and format requirements
- Create TRACKED_BY relationships linking requirements to CDRLs

**Requirements Tables (Section C):**
- Extract relationships between requirements and their specifications
- Separate workload (requirement) from surveillance (performance_standard)

**Cross-Reference Tables:**
- Extract Section L ↔ M mappings directly from table structures
- Volume/Factor correspondence tables → GUIDES relationships
- Compliance matrices → EVALUATED_BY relationships

PART C: CRITICAL DISTINCTIONS (Priority Detection Rules)
================================================================================

C.1 PERFORMANCE_STANDARD vs REQUIREMENT (Priority #1 Error)
-------------------------------------------------------------
This is the #1 extraction error. These are DIFFERENT entity types!

PERFORMANCE_STANDARD = Measurable threshold, SLA, QASP rule, or acceptance bar

Trigger Phrases - Extract as performance_standard when you see:
- "Performance Objective (PO-X)" or "PO-1", "PO-2", etc.
- "Performance Threshold:" or "Threshold:"
- "Acceptable Quality Level (AQL)"
- "Quality Assurance Surveillance Plan (QASP)"
- "Method of Surveillance:"
- Numerical standards: "99.9%", "Zero (0)", "No more than X", "At least Y%"
- "per month", "per week", "per quarter" with a standard value
- SLA, response-time, uptime, availability, error-rate, deduction language

Structure Pattern: Performance standards often appear in tables with columns:
- Requirement | Standard | Surveillance Method
- Performance Objective | Threshold | Method of Surveillance

REQUIREMENT = Action/Obligation (What contractor must DO)

Trigger Phrases - Extract as requirement when you see:
- "Contractor shall...", "Contractor must...", "Contractor will..."
- "The offeror shall...", "Personnel shall..."
- Action verbs: provide, maintain, perform, deliver, ensure, implement

NOT Performance Standards (Extract as requirement):
- "Contractor shall clean daily" → requirement (action)
- "Contractor must provide reports" → requirement (action)
- "Personnel shall complete training" → requirement (action)
- "Maintain equipment" → requirement (action without threshold)

SPLIT RULE: When a sentence contains BOTH action AND standard, extract TWO entities!

Example: "Contractor shall clean equipment daily with no more than 2 defects per month."

Extract as TWO entities:
{{"name": "Daily Equipment Cleaning", "type": "requirement", "description": "The Contractor shall clean equipment daily. Criticality: MANDATORY. Modal verb: shall."}}
{{"name": "Equipment Cleaning Defect Threshold", "type": "performance_standard", "description": "No more than 2 defects per month for equipment cleaning. Threshold: No more than 2 defects per month. Measurement: Periodic Inspection (Monthly)."}}
{{"source": "Daily Equipment Cleaning", "target": "Equipment Cleaning Defect Threshold", "keywords": "MEASURED_BY", "description": "Cleaning requirement performance is measured by defect threshold"}}

C.2 STRATEGIC_THEME + CUSTOMER_PRIORITY Detection (Shipley Capture Intelligence)
----------------------------------------------------------------------------------
Capture intelligence has TWO extraction targets:
- CUSTOMER_PRIORITY = what the customer EXPLICITLY states is important
- STRATEGIC_THEME = what the proposer INFERS from patterns in the RFP

STRATEGIC_THEME Types (ThemeType enum):
1. DISCRIMINATOR: Key differentiator between competitors (INFERRED)
2. PROOF_POINT: Evidence/past performance that validates capability (INFERRED)
3. WIN_THEME: Overarching proposal messaging/positioning (INFERRED)

*** NOTE: CUSTOMER_HOT_BUTTON is NO LONGER a STRATEGIC_THEME subtype.
    Hot buttons are now extracted as CUSTOMER_PRIORITY entities. ***

CUSTOMER_PRIORITY Detection Signals (extract as customer_priority):
- "The Government places emphasis on..."
- "Critical to mission success is..."
- "Of paramount importance..."
- "The Government's primary concern is..."
- "Essential to successful performance..."
- "Key to award decision..."
- Evaluation factor weighted >30% (signals priority)
- "Key to success is..."
- "Past performance demonstrating [X] is essential"
- Repeated emphasis on specific capability areas
- "Significantly more important than..."
- Adjectival rating comparisons between factors

DISCRIMINATOR Detection (extract as strategic_theme, theme_type=DISCRIMINATOR):
- Unique capability requirements that favor specific approaches
- Technology or methodology preferences stated in RFP
- Requirements where a specific approach would stand out
- Ghost language ("contractors without X may struggle to...")

WIN_THEME Detection (extract as strategic_theme, theme_type=WIN_THEME):
- Intersection of customer priorities + pain points + evaluation weights
- Emphasis on innovation, cost savings, or efficiency
- References to lessons learned from previous contracts

PROOF_POINT Detection (extract as strategic_theme, theme_type=PROOF_POINT):
- Specific past performance requirements
- CPARS rating references
- Certification requirements (CMMI, ISO, etc.)
- Quantified experience (years, contract values, staff size)
- Similar contract references

Example - Explicit Customer Statement → CUSTOMER_PRIORITY:
"Factor 1: Mission Readiness (50%) - Most Important"
Extract as: customer_priority
Name: "Mission Readiness Is Top Priority"

Example - Explicit Customer Emphasis → CUSTOMER_PRIORITY:
"Cultural sensitivity in host nation operations is essential to contract success."
Extract as: customer_priority
Name: "Host Nation Cultural Sensitivity Priority"

Example - Inferred Discriminator → STRATEGIC_THEME:
"Contractor must maintain TS/SCI cleared personnel on-site at all times."
Extract as: strategic_theme with theme_type DISCRIMINATOR
Name: "Cleared Personnel On-Site Requirement (DISCRIMINATOR)"

C.3 Requirement Criticality Classification
-------------------------------------------
MANDATORY (Priority Score: 100): SHALL/MUST/WILL (with contractor/offeror as subject) - Zero defects, non-negotiable compliance
IMPORTANT (Priority Score: 75): SHOULD (with contractor/offeror as subject) - High priority, best practice, expected but not absolute
OPTIONAL (Priority Score: 25): MAY (with contractor/offeror as subject) - Contractor discretion, alternative approaches acceptable
INFORMATIONAL (Priority Score: 0): SHALL/MUST with Government as subject - NOT a contractor requirement, extract as concept or skip

CRITICAL: Check the SUBJECT of the sentence!
- "Contractor shall provide..." → REQUIREMENT (MANDATORY)
- "Government shall provide..." → CONCEPT (NOT a requirement)

C.3.1 Requirement Type Classification (Content-Based)
------------------------------------------------------
Classify requirements by their functional nature (use req_type field):

FUNCTIONAL - "provide", "deliver", "perform", "execute" + service/product
   Examples: "Contractor shall provide 24/7 help desk support"
             "Perform preventive maintenance on all equipment"
             "Deliver monthly status reports"

PERFORMANCE - SLA language, metrics, measurable outcomes
   Examples: "99.9% system uptime"
             "Response within 24 hours of incident report"
             "First-time fix rate ≥ 95%"

SECURITY - NIST references, FedRAMP, clearance levels, cybersecurity
   Examples: "FedRAMP Moderate authorization required"
             "Personnel require Secret clearance"
             "Implement NIST 800-53 controls"

TECHNICAL - Technology stack, platforms, software versions, infrastructure
   Examples: "Deploy on AWS GovCloud"
             "Use Java 11 or higher"
             "PostgreSQL database required"

INTERFACE - Integration points, data exchange, APIs, system connections
   Examples: "Integrate with agency ERP system"
             "Support HTTPS/TLS 1.3"
             "RESTful API with OAuth 2.0"

MANAGEMENT - Reporting, governance, oversight, project management
   Examples: "Submit monthly status reports"
             "Use Agile/Scrum methodology"
             "PMI-certified Project Manager required"

DESIGN - Standards compliance, architectural mandates, design constraints
   Examples: "Section 508 accessibility compliance"
             "Follow agency branding guidelines"
             "Microservices architecture"

QUALITY - QA processes, testing, verification, validation
   Examples: "Automated test coverage ≥ 80%"
             "ISO 9001 certification"
             "Peer code review required"

PART C.4: PRIORITY ORDER FOR ENTITY EXTRACTION
================================================================================

When processing an RFP, prioritize extraction in this order:

1. PROGRAM - Identify top-level program FIRST (provides context for all others)
2. DOCUMENT_SECTION - Map document structure (provides location context even outside UCF)
3. EVALUATION_FACTOR + SUBFACTOR - Critical for proposal planning
4. PROPOSAL_INSTRUCTION + PROPOSAL_VOLUME - Determine proposal structure
5. REQUIREMENT - Core work obligations
6. CONTRACT_LINE_ITEM + PRICING_ELEMENT - Pricing structure and commercial logic
7. WORK_SCOPE_ITEM + TRANSITION_ACTIVITY - Work packages and phase-in/phase-out scope
8. WORKLOAD_METRIC + LABOR_CATEGORY - Quantitative BOE drivers and named labor roles
9. PERFORMANCE_STANDARD - Quality standards, thresholds, and surveillance rules
10. DOCUMENT + AMENDMENT - Structural source documents and changes
11. CLAUSE + REGULATORY_REFERENCE + TECHNICAL_SPECIFICATION - Governing authorities
12. DELIVERABLE + COMPLIANCE_ARTIFACT + PAST_PERFORMANCE_REFERENCE - Evidence and submission objects
13. GOVERNMENT_FURNISHED_ITEM - Customer-provided assets and facilities
14. STRATEGIC_THEME + CUSTOMER_PRIORITY + PAIN_POINT - Competitive intelligence
15. Standard Entities - Organizations, concepts, events, equipment, technology, etc.

**Why This Order Matters**: Early entities provide context for later entities:
- Knowing the PROGRAM helps classify requirements (program-specific vs general)
- Knowing SECTIONS helps attribute entities to their source locations
- Knowing EVALUATION_FACTORS helps identify strategic themes (weighted >30% = hot button)
- Requirements inherit context from their parent sections

PART E: COMMON SOLICITATION STRUCTURE PATTERNS (UCF IS ONE PATTERN, NOT A REQUIREMENT)
================================================================================

Use CONTENT and STRUCTURE patterns, not only UCF labels. UCF is common in federal RFPs,
but the ontology must work for RFQ, BPA call, task order, OTA, annex-heavy packages,
and badly structured attachments.

Pattern 1: Structural Containers
   Entity types: document_section, document, amendment
   Content signals: Section, Article, Paragraph, Appendix, Annex, Exhibit, Attachment,
   numbered headings, titled subsections, revision/modification notices

Pattern 2: Commercial / Pricing Content
   Entity types: contract_line_item, pricing_element, proposal_volume
   Content signals: CLIN/SLIN structure, price schedules, fee tables, escalation, rate tables

Pattern 3: Work Scope Content
   Entity types: work_scope_item, requirement, deliverable, transition_activity, workload_metric, performance_standard
   Content signals: task descriptions, objectives, performance work, shall-statements, QASP/SLA tables, transition plans

Pattern 4: Proposal Preparation Content
   Entity types: proposal_instruction, proposal_volume, evaluation_factor, subfactor, past_performance_reference
   Content signals: page limits, font/margin rules, response volume names, "will be evaluated", weighting, PPQ/CPARS instructions

Pattern 5: Governing Authorities
   Entity types: clause, regulatory_reference, technical_specification, compliance_artifact
   Content signals: FAR/DFARS clauses, DAFI/NIST references, MIL-DTL/ICD/drawing references, certifications/ATOs

Pattern 6: Resource / Ownership Content
   Entity types: government_furnished_item, equipment, technology, location, organization, person
   Content signals: GFE/GFI/GFP, facilities, systems furnished by government, asset inventories, POCs

Pattern 7: Strategic Signal
   Entity types: strategic_theme, customer_priority, pain_point
   Content signals: mission emphasis, paramount/critical language, known deficiencies, lessons learned, repeated priorities

UCF Examples (Common but NOT mandatory):
- Section A / cover page → document_section, organization, person, event
- Section B / pricing schedule → document_section, contract_line_item, pricing_element
- Section C / PWS-SOW-SOO → document_section, work_scope_item, requirement, workload_metric, performance_standard
- Section H / special requirements → document_section, requirement, labor_category, location, government_furnished_item
- Section I / clauses → document_section, clause
- Section J / attachments → document_section, document, technical_specification, deliverable
- Section L / instructions → document_section, proposal_instruction, proposal_volume
- Section M / evaluation → document_section, evaluation_factor, subfactor, strategic_theme, customer_priority

Non-UCF Label Mapping:
- "Proposal Instructions" / "Response Format" / "Submission Requirements" → proposal_instruction
- "Evaluation Criteria" / "Selection Criteria" / "Award Methodology" → evaluation_factor

Mixed Content Handling:
Some headings contain multiple entity types. Extract ALL of them.

F.1 Mandatory Relationship Types
---------------------------------
Use EXACTLY these relationship types. The relationship_type field in the output
MUST match one of these types verbatim (UPPERCASE_WITH_UNDERSCORES). Do NOT
invent new types, use free-text descriptions, or vary the casing.

─── STRUCTURAL ───
CHILD_OF - Hierarchical structure (sections, subsections, numbered documents)
ATTACHMENT_OF - Documents attached to document sections or parent documents
CONTAINS - Parent contains child
AMENDS - Amendment modifies a document, section, or requirement
SUPERSEDED_BY - Prior version replaced by a later version
REFERENCES - Entity mentions another entity by name

─── EVALUATION & PROPOSAL ───
GUIDES - Proposal instructions or volumes that guide evaluation responses
EVALUATED_BY - Requirements/deliverables/past performance/work scope items evaluated in factors
HAS_SUBFACTOR - Evaluation factor has subfactor (Factor 1 → Subfactor 1.1)
MEASURED_BY - Requirement is measured by performance standard
EVIDENCES - Past performance reference or compliance artifact evidences a theme, factor, or requirement

─── WORK & DELIVERABLES ───
PRODUCES - Work scope items or transition activities produce deliverables
SATISFIED_BY - Requirement satisfied by deliverable or capability
TRACKED_BY - Deliverable tracked by CDRL
SUBMITTED_TO - Deliverable submitted to organization
STAFFED_BY - Requirement or work scope item staffed by labor category
PRICED_UNDER - Deliverable or requirement priced under contract line item
FUNDS - Contract line item funds a requirement or deliverable
QUANTIFIES - Workload metric quantifies a service requirement

─── AUTHORITY & GOVERNANCE ───
GOVERNED_BY - Requirement governed by policy/document
MANDATES - Regulatory reference mandates a requirement
CONSTRAINED_BY - Work scope or pricing element constrained by authority/spec
DEFINES - One entity defines/explains another
APPLIES_TO - Requirement/standard applies to equipment; instruction applies to volume

─── RESOURCE & OPERATIONAL ───
HAS_EQUIPMENT - Location has equipment
PROVIDED_BY - Government furnished item is provided by organization
COORDINATED_WITH - Activity coordinated with organization
REPORTED_TO - Documentation reported to organization

─── STRATEGIC & CAPTURE ───
ADDRESSES - Requirement or solution addresses customer priority or pain point
RESOLVES - Requirement, deliverable, or transition activity resolves a pain point
SUPPORTS - One entity enables/supports another
RELATED_TO - Semantic similarity, shared topics

⛔ BANNED TYPES (these are NOT valid — NEVER use them):
belongs_to, contained_in, part_of, has, is_a, type_of, member_of, associated_with
→ Use the specific types above. See F.10 for table co-occurrence mapping rules.

F.2 Proposal Instruction ↔ Evaluation Mapping (The "Golden Thread")
--------------------------------------------------------------------
CRITICAL: Proposal instructions and evaluation factors are mirror images even when the solicitation is not UCF-organized.

Explicit Link: If instructions say "Volume 1 addresses Factor 1" → Create GUIDES relationship
Implicit Link: If proposal guidance mentions "Technical Approach" and evaluation has "Technical Factor" → Create GUIDES relationship
Page Limits: If a page limit is mentioned near a factor → Create GUIDES relationship

Pattern 1: Explicit Factor Reference
  Submission Instruction text: "Volume 2 shall address Evaluation Factor 3"
  → GUIDES relationship (confidence: 0.95)

Pattern 2: Topic Match - Technical
  Submission Instruction: "Technical Approach Volume (25 pages)"
  Evaluation Factor: "Factor 2: Technical Approach (40%)"
  → GUIDES relationship (confidence: 0.90)

Pattern 3: Topic Match - Management
  Submission Instruction: "Management Volume limited to 15 pages"
  Evaluation Factor: "Factor 3: Management Capability"
  → GUIDES relationship (confidence: 0.90)

Pattern 4: Topic Match - Past Performance
  Submission Instruction: "Past Performance Information (no page limit)"
  Evaluation Factor: "Factor 4: Past Performance (30%)"
  → GUIDES relationship (confidence: 0.90)

Pattern 5: Page Limit + Factor Co-location
  Evaluation Factor text mentions: "Technical Volume: 25 pages. Evaluates technical approach..."
  → Infer submission instruction exists, create GUIDES relationship

Pattern 6: Format Requirements Near Factor
  Text: "Proposals shall include system diagrams. Factor 1 evaluates system architecture."
  → Create GUIDES relationship between format requirement and factor

Pattern 7-20: Semantic Similarity Mapping
For ANY combination of:
- Submission Instructions: Technical Volume, Management Volume, Past Performance, Price Volume, Small Business Subcontracting Plan, Transition Plan, Quality Plan, Staffing Plan, Risk Management Approach, Security Plan
- Evaluation Factors: Technical Factor, Management Factor, Past Performance Factor, Price Factor, Small Business Factor, Transition Factor, Quality Factor, Personnel Factor, Risk Factor, Security Factor

Algorithm:
1. Extract topic keywords from instruction name (remove "Volume", "Plan", stopwords)
2. Extract topic keywords from factor name (remove "Factor", stopwords)
3. If similarity >= 0.50 (strong match): GUIDES with confidence 0.90
4. If similarity >= 0.30 (moderate match): GUIDES with confidence 0.75
5. If keyword appears in factor description: GUIDES with confidence 0.70

Pattern 21-30: Cross-Volume References
If Submission Instruction mentions multiple volumes addressing same factor:
"Technical and Management Volumes collectively address Factors 1-3"
→ Create GUIDES relationships for BOTH volumes to ALL three factors

Pattern 31-40: Subfactor Decomposition
If Factor has subfactors:
Factor 1: Technical Approach
  - Subfactor 1.1: System Architecture
  - Subfactor 1.2: Integration Methodology

And Submission Instruction mentions specific technical topics:
"Include architecture diagrams and integration plans"
→ Create GUIDES relationships to specific subfactors

Pattern 41-50: Implicit Format Instructions
When factor description implies required content:
Factor text: "Evaluate contractor's quality assurance processes and ISO 9001 certification"
→ Implied submission instruction: Quality plan must address ISO 9001
→ Create GUIDES relationship if quality submission instruction exists

F.9 Structural Bridge Rules (CRITICAL FOR GRAPH TRAVERSAL)
------------------------------------------------------------
These rules create edges that connect typed entities back to their structural
context. Without them, graph queries hit dead-ends — the knowledge exists but
the traversal path is broken.

Rule 1: Section Provenance for Non-Numbered Entities
=====================================================
PROBLEM: Entities like "Volume I Technical", "Factor 1: Technical Approach", and
"Technical Volume limited to 25 pages" do NOT have X.Y.Z numbering, so the
universal numbered hierarchy rule (F.5 Rule 6) never fires. They become orphans
disconnected from the document structure they belong to.

ACTION: When extracting proposal_volume, evaluation_factor, or proposal_instruction
entities, ALWAYS create a CHILD_OF relationship to their parent document_section.

  proposal_volume --CHILD_OF--> document_section (Section L or equivalent)
  evaluation_factor --CHILD_OF--> document_section (Section M or equivalent)
  proposal_instruction --CHILD_OF--> document_section (Section L or equivalent)

Non-UCF equivalents:
  "Proposal Instructions" / "Submission Requirements" → parent for volumes + instructions
  "Evaluation Criteria" / "Selection Criteria" / "Award Methodology" → parent for factors
  "Basis for Award" / "Source Selection" → parent for factors

EXAMPLES:
  "Volume I Technical" --CHILD_OF--> "Section L Proposal Instructions"
  "Factor 1 Technical Approach" --CHILD_OF--> "Section M Evaluation Criteria"
  "Technical Volume limited to 25 pages" --CHILD_OF--> "Section L"
  "Subfactor 1.1 TOMP" --CHILD_OF--> "Factor 1 Technical Approach" (via HAS_SUBFACTOR)

Confidence: 0.90 (high - entity appears within the section text)
WHY THIS MATTERS: Queries like "What does Section M evaluate?" cannot traverse
from document_section → evaluation_factor without this edge.

Rule 2: Proposal Instruction → Proposal Volume (APPLIES_TO)
=============================================================
PROBLEM: Instructions prescribe rules FOR volumes (page limits, formatting, what
to address). Both connect to evaluation_factor via GUIDES, but there is no direct
edge between them. To answer "What are the rules for Volume II?" requires vector
search only — the graph path is broken.

ACTION: When a proposal_instruction mentions a specific volume, create:
  proposal_instruction --APPLIES_TO--> proposal_volume

PATTERNS:
  "Technical Volume shall not exceed 25 pages" → instruction APPLIES_TO "Volume I Technical"
  "Management Volume limited to 15 pages, 12pt font" → instruction APPLIES_TO "Volume II Management"
  "Cost Volume shall include completed pricing schedules" → instruction APPLIES_TO "Cost Volume"
  "All volumes shall use 1-inch margins" → instruction APPLIES_TO EACH volume

Confidence: 0.85-0.95 depending on explicitness
WHY THIS MATTERS: The compliance matrix needs instruction→volume→factor paths.

Rule 3: Work Scope Item → Evaluation Factor (EVALUATED_BY)
============================================================
PROBLEM: RFPs say "Factor 1 evaluates the offeror's approach to tasks 3.1–3.5"
but EVALUATED_BY only connected requirements and deliverables to factors. Work
scope items (task packages) were disconnected from what factor evaluates them.

ACTION: When evaluation factor text references work scope tasks, SOW sections,
or task areas, create:
  work_scope_item --EVALUATED_BY--> evaluation_factor

PATTERNS:
  Factor text: "Technical Approach to PWS tasks 3.1 through 3.5" → each WSI EVALUATED_BY factor
  Factor text: "Maintenance operations as described in Section C.4" → WSI EVALUATED_BY factor
  Factor text: "Evaluate logistics support plan" → logistics WSIs EVALUATED_BY factor

Confidence: 0.80-0.90 depending on explicitness
WHY THIS MATTERS: "Which tasks does Technical Factor evaluate?" is critical for
Shipley compliance matrix and proposal outlining. Without this edge, the answer
requires vector search across disconnected graph neighborhoods.

F.10 Table Co-Occurrence Relationship Rules (CRITICAL FOR MULTIMODAL CONTENT)
------------------------------------------------------------------------------
PROBLEM: When MinerU/VLM processes tables, entities from the same table appear
in the same text chunk without prose connectors ("shall", "requires", "measured by").
The extraction LLM defaults to generic types like "belongs_to", "contained_in",
or "part_of" because no linguistic trigger fires from Part F.1.

╔══════════════════════════════════════════════════════════════════════════════╗
║  RULE: NEVER use "belongs_to", "contained_in", or "part_of" as a           ║
║  relationship type. These are NOT in the valid type set (Part F.1).         ║
║  Instead, use the entity-pair mapping below.                                ║
╚══════════════════════════════════════════════════════════════════════════════╝

When two entities co-occur in table-derived text (marked by workload data,
tabular structure, or column/row patterns), use these typed relationships
based on the entity pair:

─── WORK & DELIVERABLES ───
requirement ↔ deliverable         → SATISFIED_BY
requirement ↔ performance_standard → MEASURED_BY
requirement ↔ workload_metric     → QUANTIFIES
work_scope_item ↔ deliverable     → PRODUCES
deliverable ↔ contract_line_item  → PRICED_UNDER (deliverable→CLIN) or FUNDS (CLIN→deliverable)
requirement ↔ contract_line_item  → PRICED_UNDER (req→CLIN) or FUNDS (CLIN→req)

─── EVALUATION ───
requirement ↔ evaluation_factor   → EVALUATED_BY
deliverable ↔ evaluation_factor   → EVALUATED_BY
work_scope_item ↔ evaluation_factor → EVALUATED_BY

─── STRUCTURAL ───
entity ↔ document_section         → CHILD_OF (entity→section) or CONTAINS (section→entity)
entity ↔ document                 → CHILD_OF (entity→document) or CONTAINS (document→entity)

─── AUTHORITY ───
requirement ↔ clause              → GOVERNED_BY (req→clause) or MANDATES (clause→req)
requirement ↔ regulatory_reference → GOVERNED_BY (req→reg) or MANDATES (reg→req)

─── RESOURCE ───
requirement ↔ labor_category      → STAFFED_BY
work_scope_item ↔ labor_category  → STAFFED_BY
location ↔ equipment              → HAS_EQUIPMENT
deliverable ↔ organization        → SUBMITTED_TO

─── STRATEGIC ───
requirement ↔ customer_priority   → ADDRESSES
requirement ↔ pain_point          → RESOLVES

DEFAULT: If none of the above pairs match, use RELATED_TO (not belongs_to).

EXAMPLE (Table-derived text):

PART G: ENTITY NAMING NORMALIZATION RULES
================================================================================

CRITICAL: Prevent duplicate entities due to formatting variations!

Government RFPs use inconsistent formatting:
- Page titles: "SECTION C.4 - SUPPLY" (ALL CAPS)
- Text references: "Section C.4" (Title Case)
- Inline mentions: "section c.4" (lowercase)
- Cross-references: "Sec C.4", "C.4", "Section C-4"

YOU MUST extract these as ONE entity, not multiple!

Normalization Rules:

1. Section Names: Always use Title Case with periods
   ✅ CORRECT: "Section C.4 Supply"
   ❌ WRONG: "SECTION C.4", "section c.4", "Sec C.4", "Section C-4"

2. FAR/DFARS Clauses: Always use exact citation format
   ✅ CORRECT: "FAR 52.212-1"
   ❌ WRONG: "far 52.212-1", "FAR 52.212.1", "FAR52.212-1"

3. CDRL/Deliverables: Always use uppercase identifier + descriptive name
   ✅ CORRECT: "CDRL A001 Monthly Status Report"
   ❌ WRONG: "CDRL a001", "Cdrl A001", "cdrl A001"

4. Organizations/Programs: Use official capitalization from context
   ✅ CORRECT: "Marine Corps Prepositioning Program" or "MCPP II"
   ❌ WRONG: "MARINE CORPS PREPOSITIONING PROGRAM", "mcpp ii"

5. When you see multiple formatting variations:
   - Identify they refer to the same entity
   - Extract ONCE using the most complete, properly formatted version
   - Merge descriptions from all mentions

Example - Section with Multiple Formats:
Text contains:
- Page 15 title: "SECTION C.4 - SUPPLY"
- Page 27 reference: "per Section C.4"
- Page 45 mention: "section c.4 requirements"

Extract ONCE as:
{{"name": "Section C.4 Supply", "type": "section", "description": "SECTION C.4 - SUPPLY. This section defines materiel requirements and supply chain specifications for the contract."}}

NOT three separate entities!

Example - Clause with Variations:
Text contains:
- "FAR 52.212-1 Instructions to Offerors"
- "far 52.212-1"
- "FAR clause 52.212-1"

Extract ONCE as:
{{"name": "FAR 52.212-1", "type": "clause", "description": "FAR 52.212-1 Instructions to Offerors—Commercial Products and Commercial Services (SEP 2023). Clause number: 52.212-1. Regulation: FAR."}}

PART H: DECISION TREE FOR AMBIGUOUS CASES

Q: Document with "J-02000000 Performance Work Statement" - What entity type?
A: Extract as BOTH:
   - document: "J-02000000 Performance Work Statement" (for section linkage)
   - work_scope_item: "Performance Work Statement" (for semantic content)
   Why: Preserves both structural (attachment) and semantic (work definition) relationships

Q: Text says "Clean daily; no more than 2 errors per month" - One entity or two?
A: TWO entities!
   - requirement: "Daily Cleaning" (The work to be done)
   - performance_standard: "No more than 2 errors per month" (The standard)
   Why: Workload (cost driver) must be separated from Surveillance (risk driver)

Q: Text says "Contractor shall have 5 years experience" - REQUIREMENT or EVALUATION_FACTOR?
A: REQUIREMENT (experience requirement)
   - If also mentioned in Section M, create second entity as EVALUATION_FACTOR
   - Link them: REQUIREMENT --EVALUATED_BY--> EVALUATION_FACTOR
   Why: Requirements and evaluation are different concepts (what vs how scored)

Q: Clause text "The Government shall provide GFE within 30 days" - Extract as REQUIREMENT?
A: NO - Skip government obligations
   - Only extract contractor obligations (shall/must/should/may)
   - Government obligations are informational context, not requirements
   Why: Requirements = contractor obligations only

Q: Text says "CDRL A001 Monthly Status Report" in both Section C and Section J?
A: Extract ONCE as deliverable with full identifier "CDRL A001"
   - Link to both source sections if mentioned in multiple locations
   - Avoid duplicate entities (same CDRL = same deliverable)
   Why: Physical deliverables are unique, even if referenced multiple times

Q: "Navy MBOS Program" vs "Concorde RG-24 Battery" - How to distinguish?
A: Look for model numbers and physical characteristics:
   - program: Services, initiatives, no model numbers (Navy MBOS, MCPP II)
   - equipment: Model numbers, physical items (Concorde RG-24, 6200 Tennant)
   Why: Programs are abstract services; equipment is tangible hardware

Q: Section H says "Contractor shall maintain security clearances" - What type?
A: requirement (special requirement from Section H)
   - Section H requirements are still requirements (location doesn't change type)
   - Extract modal verb (shall) and subject (security clearances)
   Why: Content-based typing, not location-based

Q: "Exhibit A - Pricing Schedule" - DOCUMENT or something else?
A: DOCUMENT (pricing attachment)
   - Extract individual CLINs/SLINs as CONTRACT_LINE_ITEM entities
   - Link to Section B (Supplies/Prices) with ATTACHMENT_OF relationship
   Why: Pricing exhibits are documents containing structured line items

Q: Text references "MIL-STD-882E" but document not attached?
A: REGULATORY_REFERENCE (referenced standard)
   - Extract with description noting it's referenced (not attached)
   - Create REFERENCES relationship from citing document section
   Why: Referenced standards are still entities even when external

PART J: OUTPUT FORMAT (LightRAG Native JSON Structured Output)
================================================================================

CONTRACT: Return ONE valid JSON object with TWO top-level arrays — `entities`
and `relationships`. NO markdown code fences. NO commentary before or after.
NO trailing text. The HTTP layer requests `response_format={{"type":"json_object"}}`,
so anything that is not a valid JSON object will be rejected by the server.

TOP-LEVEL SHAPE:
{{
  "entities": [ ... ],
  "relationships": [ ... ]
}}

ENTITY OBJECT (3 required fields):
{{
  "name":        "<canonical entity name, Title Case for proper nouns>",
  "type":        "<one of the 33 valid entity types, lowercase_with_underscores>",
  "description": "<full description with ALL type-specific metadata embedded>"
}}

RELATIONSHIP OBJECT (4 required fields):
{{
  "source":      "<source entity name, must match an entity in `entities`>",
  "target":      "<target entity name, must match an entity in `entities`>",
  "keywords":    "<CANONICAL_TYPE[, optional, semantic, keywords]>",
  "description": "<complete sentence explaining the relationship>"
}}

CRITICAL: `keywords` MUST begin with EXACTLY ONE canonical UPPERCASE relationship
type as its first comma-separated token. Optional additional semantic keywords
MAY follow AFTER A COMMA (never a space). Examples:
  "keywords": "GUIDES"
  "keywords": "GUIDES, golden thread, L↔M mapping"
  "keywords": "CHILD_OF"
  "keywords": "MEASURED_BY, threshold, monthly inspection"

NEVER concatenate two canonical types with a space (or any other separator) in a
single `keywords` field. If two distinct canonical relationships exist between
the same source and target, emit TWO separate relationship objects — one per
canonical type. Examples of FORBIDDEN keywords (DO NOT EMIT):
  "keywords": "CONTAINS PHASE_OF"        ← space-joined, WRONG
  "keywords": "REQUIRES USES"            ← space-joined, WRONG
  "keywords": "INCLUDES SEQUENCE"        ← space-joined, WRONG
  "keywords": "VALIDATES, REVIEWS"       ← comma-joined CANONICAL types, WRONG (split into 2 relationships)
  "keywords": "CHILD_OF, CONTAINS"       ← comma-joined CANONICAL types, WRONG (split into 2 relationships)

The first token is parsed by downstream inference as the typed edge label and
MUST match the regex `^[A-Z][A-Z0-9_]*$`. Extra tokens after the first comma
enrich VDB embedding semantics (lowercase phrases, free text) but never override
the type and MUST NOT themselves be canonical UPPERCASE types.

VALID ENTITY TYPES (33 — see Part D / {entity_types_guidance}):

organization, technology, concept, location, event, person
