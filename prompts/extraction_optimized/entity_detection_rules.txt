Role: Semantic-first entity extraction for federal RFPs. Content determines entity type, not section labels. Use in RAG-Anything.

Rules:
- Prioritize correct detection of performance_metric vs requirement.
- Apply Split Rule when action + metric appear together.
- Prefer content signals over section labels; map non-standard labels to UCF equivalents.

Types (focus in Batch 1):
- performance_metric
- requirement
- strategic_theme
- section, statement_of_work, clause, document
- evaluation_factor, submission_instruction

Performance Metric Detection:
- Trigger phrases: Performance Objective (PO-X); Threshold; Method of Surveillance; Acceptable Quality Level (AQL); QASP rows; numerical standards with timeframe; "No more than X per [timeframe]"; "At least X%"; "Zero (0) [violations/errors/discrepancies]".
- Extract all above as performance_metric.
- NOT metrics: Action-only statements (contractor shall/must; personnel shall; maintain/provide) → requirement.

Split Rule (Combined Sentences):
- Input: "Contractor shall provide 24/7 support with 99.9% uptime."
- Output:
  - requirement: "24/7 Support Provision"
  - performance_metric: "99.9% Uptime Standard"
  - Relationship: requirement --MEASURED_BY--> performance_metric

Strategic Themes (Shipley Capture Intelligence):
- Theme types:
  - CUSTOMER_HOT_BUTTON: Government explicit priority/concern (emphasizes, critical to, paramount, essential, heavy weighting)
  - DISCRIMINATOR: Differentiator between competitors (unique reqs, specialized capabilities, innovation)
  - PROOF_POINT: Evidence of capability (past performance, certifications, demonstrated experience)
  - WIN_THEME: Proposal positioning (alignment language, value proposition)
- Customer Hot Button signals:
  - Emphasis language: "Government places emphasis", "Critical to mission success", "Paramount importance", "Primary concern", "Essential to successful performance", "Key to award decision"
  - Weighting: e.g., factors >30%, "Technical twice as important as price", "Past Performance significantly more important than cost"
  - Repetition: capability mentioned 3+ times across sections; appears in both PWS and evaluation factors; special emphasis sections (e.g., H)
- Examples:
  - "Factor 1: Mission Readiness (50%) - Most Important" → strategic_theme, theme_type "CUSTOMER_HOT_BUTTON", name "Mission Readiness Priority"
  - "Cultural sensitivity... essential to contract success." → strategic_theme, theme_type "CUSTOMER_HOT_BUTTON", name "Host Nation Cultural Sensitivity"
  - "Offerors must demonstrate relevant experience in austere environments." → strategic_theme, theme_type "CUSTOMER_HOT_BUTTON", name "Austere Environment Experience Requirement"

Semantic-First Detection Principle:
- Traditional (fragile): IF section_label == "Section L" → entity_type "section".
- Adaptive (correct): IF evaluation criteria language → entity_type "evaluation_factor"; also capture section_origin via context.
- Rationale: RFP structures vary; content signals are universal, labels are not.

UCF Mapping (content-first guidance):
- Section A: solicitation/contract form → section, organization, person, event; signals: SF 1449, solicitation number, POCs, Q&A deadline.
- Section B: supplies/services & prices → section, concept, program; signals: CLIN/SLIN, pricing tables, line items, unit prices.
- Section C: statement of work → section, statement_of_work, requirement, deliverable, performance_metric; signals: "contractor shall", tasks, performance objectives, QASP.
- Section H: special requirements → section, requirement, person, location; signals: clearances, key personnel, conflicts, facility requirements.
- Section I: contract clauses → section, clause; signals: FAR/DFARS patterns "52.###-##", "incorporated by reference".
- Section J: attachments → section, document, statement_of_work; signals: "Attachment", "Annex", "Exhibit", "J-####".
- Section L: instructions to offerors → section, submission_instruction; signals: page limits, fonts, proposal shall, volumes, deadlines.
- Section M: evaluation factors → section, evaluation_factor, strategic_theme; signals: "will be evaluated", factor hierarchy, relative importance, adjectival ratings.

Non-Standard Label Mapping:
- RFQ → Section A (cover signals)
- Technical Requirements / Work Requirements / SOO → Section C
- Proposal Instructions / Submission Requirements → Section L
- Selection Criteria / Evaluation Methodology / Source Selection Plan → Section M

Agency-Specific Terms:
- Scope of Work / PWS → statement_of_work
- Special Contract Requirements → requirement
- Applicable Clauses → clause
- Required Certifications → requirement (compliance)

Content-Based Detection Signals:
- evaluation_factor: "will be evaluated", "evaluation factor", "most important", "adjectival rating", "source selection", "significantly more important"; numbering M1/M2, points.
- submission_instruction: "page limit", "font size", "proposal shall", "volume structure"; submit by, electronic submission, format requirements; e.g., "Times New Roman 12pt", 1-inch margins.
- clause: FAR/DFARS/AFFARS/NMCARS patterns ("52.###-##", "252.###-####"), "incorporated by reference", clause title, full text.
- statement_of_work: tasks, objectives, deliverables; "contractor shall perform"; numbering 1.0/1.1; labels SOW/PWS/SOO.

Mixed Content Handling:
- Example: Section M includes embedded instructions → extract both evaluation_factor and submission_instruction; link SUBMISSION_INSTRUCTION --EVALUATED_BY--> EVALUATION_FACTOR.
- Example: Section C references Attachment → extract SECTION, DOCUMENT, STATEMENT_OF_WORK; link SECTION --REFERENCES--> DOCUMENT --CONTAINS--> STATEMENT_OF_WORK.

Requirements: Criticality & Classification
- CRITICAL RULE: Determine subject of sentence.
- Criticality levels:
  - MANDATORY (Priority 100): "Contractor shall", "Offeror must", "The contractor shall" (subject is contractor/offeror).
  - INFORMATIONAL (Priority 0): "Government will/shall" (subject is government/agency; not a contractor requirement).
  - IMPORTANT (Priority 75): "should", "encouraged to", "preferred", "desirable" with contractor subject.
  - OPTIONAL (Priority 25): "may", "can", "has the option to" with contractor subject.
- Type classification by content signals:
  - FUNCTIONAL: provide/deliver/perform/execute + service/product.
  - PERFORMANCE: SLA language, measurable outcomes.
  - SECURITY: NIST/FedRAMP/clearances/cyber.
  - TECHNICAL: tech stack/platforms/versions/infrastructure.
  - INTERFACE: integration/data exchange/APIs.
  - MANAGEMENT: reporting/governance/PM practices.
  - DESIGN: standards/design constraints/architecture.
  - QUALITY: QA/testing/verification/validation.
- Basic attributes: entity_name, entity_type, criticality, modal_verb, labor_drivers, material_needs.
- Example (MANDATORY FUNCTIONAL): "Contractor shall provide 24/7 help desk support" → requirement with criticality MANDATORY, modal "shall", labor_drivers including 24/7 coverage.

Program Entity:
- Purpose: top-level container for acquisition program/initiative/system.
- Signals: program names in title, major acquisition titles, portfolio initiatives, system names.
- Naming: full names, acronyms (e.g., MCPP II, Navy MBOS), system names.
- Hierarchy: program CONTAINS requirements, deliverables, technologies, statement_of_work, sections.
- Relationships: SECTION(C) --CONTAINS--> PROGRAM; PROGRAM --CONTAINS--> REQUIREMENT/TECHNOLOGY/DELIVERABLE.
- Detection: appears in document title, Section C/SOW intro, emphasized as umbrella subject, referenced throughout.
- Basic attributes: entity_name, acronym, scope.

Statement of Work (SOW/PWS/SOO):
- Semantic equivalence: treat SOW, PWS, SOO as STATEMENT_OF_WORK with differing styles:
  - SOW: prescriptive HOW; detailed tasks.
  - PWS: performance-based WHAT outcomes; metrics.
  - SOO: objective-based WHY; contractor derives PWS.
- Content signals: tasks, performance objectives, deliverables, scope, outcomes.
- Structure examples:
  - SOW: Task → Subtask hierarchy.
  - PWS: Objective → Performance Standard → Metric hierarchy.
  - SOO: Goal → Objective → Success Criterion hierarchy.
- Identifiers: labels (PWS/SOW/SOO), Section C, attachment references (e.g., J-#### PWS).
- Location: primary Section C; alternate attachment; embedded subsections.
- Detection focus: content (scope/tasks/objectives/deliverables), hierarchy/numbering, performance standards/metrics, deliverable schedules.
- Basic attributes: entity_name, work_type, location, performance_standards.

Annex/Attachment:
- Naming patterns: J-######, "Attachment #", "Annex ##", "Appendix X", letter-number like A-1234567.
- Parent section mapping: J-#### → Section J; Attachment # → Section J; Annex ## → Section J or standalone; letter prefix maps to section.
- Content types: may contain SOW/PWS, technical specs, maps/diagrams, data sheets, sample docs, clauses.
- Basic attributes: prefix_pattern, content_type, parent_section.
- Example: Section J lists multiple attachments; extract each as document with content_type and parent_section.

Clause Entities:
- Recognize across 26+ agency supplements.
- Patterns: FAR 52.###-##; DFARS 252.###-####; AFFARS 5352.###-##; NMCARS 5252.###-####; plus HSAR, DOSAR, GSAM, VAAR, DEAR, NFS, AIDAR, CAR, DIAR, DOLAR, EDAR, EPAAR, FEHBAR, HHSAR, HUDAR, IAAR, JAR, LIFAR, NRCAR, SOFARS, TAR.
- Extract supplement from clause number (e.g., DFARS from 252.204-7012).
- Cluster by parent section: FAR clauses → Section I; Representations/certifications → Section K.
- Group similar clauses (supplies/services, delivery, security) even if scattered.
- Basic attributes: clause_number, regulation/supplement, date.

Section Entities:
- Extract both structural label and semantic type.
- Standard UCF mapping: A→SOLICITATION_FORM, B→SUPPLIES_SERVICES, C→DESCRIPTION_SPECS, H→SPECIAL_REQUIREMENTS, I→CONTRACT_CLAUSES, J→ATTACHMENTS, K→REPRESENTATIONS, L→SUBMISSION_INSTRUCTIONS, M→EVALUATION_CRITERIA.
- Non-standard labels map: Selection Criteria→M, Technical Requirements→C, Proposal Instructions→L, RFQ→A, Statement of Work→C.
- Mixed content: sections can contain multiple types; record semantic_type and also_contains (e.g., Section M contains submission_instruction).
- Basic attributes: structural_label, semantic_type.

Strategic Themes:
- Purpose: capture competitive intelligence (priorities, emphasis, positioning).
- Types:
  - CUSTOMER_HOT_BUTTON: explicit/implicit priorities; signals include critical/essential/paramount/emphasize; factor weighting >30%; repetition across sections.
  - DISCRIMINATOR: differentiators (unique certifications, rare capabilities, innovation, incumbent knowledge, OEM partnerships, FedRAMP High, etc.).
  - PROOF_POINT: evidence (past performance, CPARS, certifications, quantified experience, similar contracts).
  - WIN_THEME: synthesis of discriminator + proof → customer benefit.
- Extraction examples: readiness priority, cybersecurity emphasis, uptime reliability, austere environment experience, cultural competence.
- From evaluation weighting: when factors reveal priorities, extract strategic_theme alongside evaluation_factor.

Section Attribution:
- Clause → Section I; representations/certifications → Section K.
- Annex prefixes: J-######, Attachment #, Annex ##, letter-prefix maps (A-#### → Section A).
- Requirement attribution: default Section C unless labeled Section H/I/K accordingly.

Submission Instructions (Embedded + Standard):
- Detect submission_instruction via: "page limit", "font size", "margins", "format requirements", "volume structure", "technical proposal shall address", "management volume limited to", submission deadlines, electronic vs hard copy, organization (Volume I/II), file types.
- CRITICAL embedded pattern: When instructions appear within evaluation factor descriptions (common non-standard case), extract both entities and link: SUBMISSION_INSTRUCTION --EVALUATED_BY--> EVALUATION_FACTOR.
- Example: "Technical Volume shall be limited to 25 pages, 12pt Times New Roman" → submission_instruction; link to nearby factor.

Evaluation Factors (Signals & Structure):
- Content signals: "will be evaluated", "evaluation factor", "adjectival rating", "scoring methodology", "most important", "significantly more important than price".
- Structure: hierarchy Factor → Subfactor → Sub-subfactor; numbering patterns M1/M2/M2.1/M2.1.1; ratings: Excellent/Good/Acceptable/Marginal/Unacceptable.
- Context: near importance statements, point scores, tradeoff descriptions; primary Section M, alternate Section L, non-standard labels like Selection Criteria.
- Example (points): "Technical Merit (100 points)" with subfactor weights; extract evaluation_factor with subfactors and weights.

Mixed Content Handling (Section-Level):
- Section M containing both evaluation criteria and embedded submission requirements → extract both; link instructions to evaluated factors.
- Section C referencing attachment for PWS → extract SECTION, DOCUMENT, STATEMENT_OF_WORK; link SECTION --REFERENCES--> DOCUMENT --CONTAINS--> STATEMENT_OF_WORK.

Deliverables:
- Signals: CDRL refs (e.g., CDRL A001, DD Form 1423), report types (monthly/quarterly/final), documentation (SDD, manuals, user guides), work products, submission language (shall submit/deliver/provide).
- Structure: CDRL numbering (A001, B012, 6022), series references (e.g., 8000 Series), explicit labels (Deliverable 1.2).
- Context: near submission schedules, lists of deliverables, SOW task references, DD Form 1423.
- Location: primary Section J attachments; alternate Section C; embedded in requirements text.
- Basic attributes: cdrl_id, due_date, format.

Document Entities:
- Signals: attachments/annex/exhibit references; referenced documents; standards; forms; prior documents/contract numbers.
- Structure: attachment numbering (J-####, Attachment #, Annex ##), standard identifiers (MIL-STD-###, ISO ####, NIST SP ###-##), form numbers (SF ###, DD ###).
- Context: "See Attachment X", "incorporated by reference", listed in Section J, reference documents sections.
- Location: primary Section J; alternate throughout the RFP; embedded in requirements citing standards.
- Basic attributes: document_number, reference_location.

Concept Entities:
- Signals: budget/pricing (CLIN/SLIN, base/option periods), technical/business concepts (Agile, DevSecOps, Zero Trust), processes (change control, risk management), abstract ideas (readiness), definitions.
- Structure: CLIN/SLIN numbering patterns; abstract nouns; process names; methodologies/frameworks.
- Differentiation:
  - Not concept if physical item (equipment), scheduled activity (event), named unit (organization), or software with version (technology).
  - Is concept if abstract idea/process, business term, CLIN/budget line item, or glossary term.
- Basic attributes: concept_category, definition.

Equipment Entities:
- Signals: model numbers; military equipment codes (TAMCN/NSN); physical assets; vehicles; hardware.
- Structure: model alphanumerics; TAMCN format; NSN format; make/model patterns.
- Context: near maintenance language; equipment lists/inventories; quantities; physical specifications.
- Differentiation: equipment (physical item) vs technology (software/system/platform).
- Basic attributes: model_number, manufacturer, specifications.

Technology Entities:
- Signals: software, platforms, systems, protocols, standards.
- Structure: version numbers; acronyms; platform names.
- Context: near technical requirements, architecture descriptions, compatibility statements.

Organization Entities:
- Signals: companies, agencies, military units, commands, departments.
- Structure: acronyms (NAVAIR, SPAWAR, GSA, DLA), corporate suffixes (Inc., LLC, Corp.), military designations (I MEF), command structures.
- Context: organizational roles, past performance, teaming arrangements, POC affiliations.
- Basic attributes: acronym, full_name, org_type.

Event Entities:
- Signals: milestones (kickoff, final delivery, award), scheduled activities (monthly reviews), deadlines (Q&A date, proposal due), periods (base/option years), processes with timing.
- Structure: dates, recurring patterns, event + timing phrases.
- Context: calendar dates, time-bound language, scheduling sections, milestone charts.
- Differentiation: event (scheduled/time-bound) vs concept (process without schedule).
- Basic attributes: event_date, frequency, milestone_type.

Person & Location Entities:
- Person signals: names, titles, roles, contact info.
- Person context: contact/POC mentions, signature blocks, key personnel sections.
- Location signals: bases, cities/states, facilities, geographic areas (CONUS/OCONUS), countries.
- Location context: performance locations, delivery addresses, site visits, geographic scoping.
- Basic attributes: for PERSON (title, role, contact_info); for LOCATION (address, facility_type).

Priority Order for Extraction:
1) PROGRAM → 2) SECTION → 3) EVALUATION_FACTOR → 4) SUBMISSION_INSTRUCTION → 5) REQUIREMENT → 6) STATEMENT_OF_WORK → 7) ANNEX → 8) CLAUSE → 9) STRATEGIC_THEME → 10) Standard entities (organization, concept, event, etc.).
- Rationale: early context improves classification and attribution of subsequent entities.

Quality Checks:
- No orphan entities; consistent naming (canonicalization); metadata completeness; subject validation (Government shall ≠ contractor requirement); prioritize content over labels.
- Entity extraction validation: required fields present; format validation (ISO dates, numeric values, booleans, arrays); cross-reference integrity (factor ↔ instruction, requirement ↔ evaluation factor, annex ↔ section, clause ↔ section); enum validation (criticality_level, theme_type, modal_verb, work_type, agency_supplement); consistency across chunks (section numbering, unique requirement IDs, modal vs criticality alignment).
