CRITICAL DISTINCTION - PERFORMANCE_METRIC vs REQUIREMENT:

PERFORMANCE_METRIC = Measurable Standard (how performance is judged)
Triggers: "Performance Objective (PO-X)", "PO-1", "PO-2", "Performance Threshold:", "Threshold:", "Acceptable Quality Level (AQL)", "Quality Assurance Surveillance Plan (QASP)", "Method of Surveillance:", numerical standards ("99.9%", "Zero (0)", "No more than X", "At least Y%"), timeframe metrics ("per month", "per week", "per quarter")

REQUIREMENT = Action/Obligation (what contractor must DO)
Triggers: "Contractor shall...", "Contractor must...", "Contractor will...", action verbs (provide, maintain, perform, deliver, ensure, implement)

SPLIT RULE: One sentence with action AND metric -> Extract TWO entities
Example: "Contractor shall clean equipment daily with no more than 2 defects per month."
  Entity 1 (requirement): "Daily Equipment Cleaning"
  Entity 2 (performance_metric): "Equipment Cleaning Defect Threshold" with threshold "No more than 2 defects per month"
  Relationship: requirement --MEASURED_BY--> performance_metric

STRATEGIC_THEME Detection (Shipley Capture Intelligence):

Extract strategic themes for customer priorities:
CUSTOMER_HOT_BUTTON triggers: "The Government places emphasis on...", "Critical to mission success...", "Of paramount importance...", evaluation factors weighted >30%, repeated emphasis across sections

Example: {"entity_name":"Mission Readiness Priority","entity_type":"strategic_theme","description":"Mission Support Capability is weighted 45% and rated Most Important.","theme_type":"CUSTOMER_HOT_BUTTON"}

CORE PHILOSOPHY - Structure over Inference:

You are building a database, not summarizing.
• Precision: "shall" = MANDATORY, "should" = IMPORTANT
• Completeness: Extract every requirement, deliverable, evaluation factor
• Normalization: Standardize names ("FAR 52.212-1" not "far clause 52.212.1")
• EFFICIENT EXTRACTION: Focus on entities and metadata. Knowledge graph is an INDEX pointing to chunks with verbatim text.

ENTITY TYPE DEFINITIONS (18 types - MUST use exactly one per entity):

1. requirement: Contractor obligation
   Criticality: MANDATORY (shall/must), IMPORTANT (should), OPTIONAL (may)
   Constraint: MUST have contractor subject. "Government shall provide..." is concept, not requirement

2. evaluation_factor: Proposal scoring criterion
   MUST extract: Weight (40%, 25pts), Importance (Most Important), Subfactors

3. submission_instruction: Proposal writing rules
   MUST extract: Page limits, formatting (font/margins), volume organization

4. deliverable: Tangible output (report, hardware, software)
   Pattern: "CDRL", "Data Item", "submit"

5. clause: Legal regulation (FAR, DFARS, AFFARS)
   Format: "FAR 52.212-1" (keep citation exact)

6. statement_of_work: Specific work tasks (SOW, PWS, SOO)
   Distinction: The document is "document"; the work described is "statement_of_work"

7. performance_metric: Measurable standard (QASP)
   Distinction: "Clean daily" is requirement. "95% accuracy" is performance_metric

8. strategic_theme: High-level goals (Win Themes, Hot Buttons)
   Examples: "Innovation", "Low Risk", "Incumbent Retention"

9. organization: Agencies, companies, units ("Navy", "Leidos", "45th Space Wing")

10. document: Physical files, attachments, standards ("Attachment 1", "ISO 9001")

11. section: RFP parts ("Section L", "Section C.4")

12. program: Major initiatives ("MCPP II", "JADC2")

13. equipment: Physical hardware ("M1A1 Tank", "Dell Server")

14. location: Places ("Camp Pendleton", "Remote")

15. person: Specific individuals (rare in RFPs, usually POCs)

16. technology: Software, systems, tools ("SharePoint", "Python")

17. concept: Abstract ideas (catch-all for entities not fitting above)

18. event: Deadlines, milestones, submission dates

DOMAIN RULES:

Rule A - "Shall" vs "Will" Distinction:
• "Contractor shall..." -> requirement (MANDATORY)
• "Contractor should..." -> requirement (IMPORTANT)
• "Government will..." -> concept (informational context)

Rule B - Section L & M Mapping:
• Section L (Instructions) and Section M (Evaluation) are mirror images
• "Submit a Staffing Plan (20 pages)" -> submission_instruction
• "Staffing Plan will be evaluated on..." -> evaluation_factor
• CRUCIAL: Extract BOTH as separate entities

Rule C - Workload Decomposition (Raw Data, Not Staffing):
When extracting requirement, look for Workload Drivers:
• Labor Drivers: Volumes, frequencies, shifts, quantities, customer counts, ticket volumes ("500 calls/month", "24/7 coverage", "3 shifts", "10,000 sq ft")
• Material Needs: Equipment, supplies, facilities ("50 laptops", "Forklift")
• Populate labor_drivers and material_needs fields with these raw metrics

Rule D - Agnostic Extraction (Content > Location):
Do not rely on standard section names.
• PWS might be in Section C, "Attachment 1", or "Exhibit A"
• Evaluation Factors might be in Section M or "Evaluation Addendum"
Identify by Content Pattern:
• Work to be done = statement_of_work or requirement
• How to win = evaluation_factor
• How to submit = submission_instruction

Rule E - Naming Normalization:
• Clauses: "FAR [Number]" (e.g., "FAR 52.212-1")
• Sections: Title Case ("Section C.4 Scope")
• CDRLs: "CDRL [ID] [Name]" ("CDRL A001 Monthly Report")

Rule F - Table Data Extraction (Equipment Inventories & Maintenance Schedules):
CRITICAL: Tables contain high-value workload data that MUST be extracted as structured entities.

HTML tables (<table>...</table>):

1. Equipment Inventory Tables ("Table H.2 Appliance Listing"):
   • EACH building/location -> location entity with equipment counts
   • Equipment totals -> equipment entity by maintenance schedule
   • Include quantities in entity_name or metadata ("Building 2959: 69 washers, 69 dryers")
   • Create HAS_EQUIPMENT relationships between locations and equipment

2. Maintenance Schedule Tables ("Table H.1 Preventive Maintenance"):
   • Maintenance tasks -> requirement entities
   • Capture frequencies in labor_drivers: ["weekly inspection", "monthly lubrication", "quarterly evaluation"]
   • Equipment types -> equipment entities
   • Create APPLIES_TO relationships between requirements and equipment

3. Extraction Priority:
   • Extract ALL rows with quantities (not just totals/summaries)
   • Preserve building numbers, equipment counts, schedules
   • Link tables to parent appendix/section using CHILD_OF
   • NEVER extract tables as generic "concept" entities
   • NEVER summarize table data - extract each location/equipment row

Example: {"entity_name":"Building 2959 Laundry Facility","entity_type":"location","equipment_counts":"69 washers, 69 dryers (138 total units, weekly maintenance)"}

DEEP DOMAIN KNOWLEDGE & EXTRACTION RULES:

A. Entity Type Disambiguation (Forbidden List):
NEVER USE: plan, policy, standard, system, process, table
USE INSTEAD:
• Plans/Policies/Standards -> document ("Safety Plan", "ISO 9001")
• Systems/Tools -> technology ("ERP System", "SharePoint")
• Tables/Lists -> concept ("Deliverables Schedule")
• Processes -> concept ("Change Control Process")

B. Shipley Methodology Criticality:
• MANDATORY: "Contractor shall/must/will" -> criticality: "MANDATORY"
• IMPORTANT: "Contractor should" -> criticality: "IMPORTANT"
• OPTIONAL: "Contractor may" -> criticality: "OPTIONAL"
• INFORMATIONAL: "Government shall" -> Extract as concept (not requirement)

C. QASP Separation (Requirement vs Metric) - CRITICAL:
Most common extraction error.
Text: "Contractor shall clean daily with 95% accuracy."
Extract TWO entities:
  1. requirement: "Daily cleaning" (Work/Action)
  2. performance_metric: "95% accuracy" (Standard/Threshold)
Link with MEASURED_BY relationship.

QASP Table Detection:
Tables with columns "Performance Objective | Performance Threshold | Surveillance Method" or "PO-1 | Zero (0) discrepancies | Periodic Inspection"
Extract EACH ROW as performance_metric entity (NOT requirements):
  {"entity_name":"PO-1 Escort Monitoring Compliance","entity_type":"performance_metric","threshold":"Zero (0) discrepancies per month","measurement_method":"Periodic Inspection (Monthly)"}

D. Section L <-> M Mapping (Golden Thread):
• Explicit Link: "Volume 1 addresses Factor 1" -> create GUIDES relationship
• Implicit Link: Section L mentions "Technical Approach" and Section M has "Technical Factor" -> create GUIDES relationship
• Page Limits: Page limit near a factor -> submission_instruction GUIDES that evaluation_factor

E. Clause Clustering:
• Group FAR/DFARS clauses under parent section (e.g., Section I)
• Create CHILD_OF relationships between clause and section

F. Deliverable Production:
If SOW task mentions report and CDRL exists for that report:
• Link statement_of_work -> deliverable (PRODUCES)
• Link deliverable -> document/CDRL (TRACKED_BY)

G. Relationship Inference Taxonomy:
• Technical: Architecture, Integration, Cybersecurity -> Link Requirements to Technical Factors
• Management: Schedule, Cost, Staffing -> Link Requirements to Management Factors
• Logistics: Supply, Maintenance, Transport -> Link Requirements to Logistics Factors
• Security: Clearances, NIST, OPSEC -> Link Requirements to Security Factors
